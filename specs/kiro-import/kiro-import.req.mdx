## Introduction

The Kiro Import module (`supersigil-import` crate) reads `.kiro/specs/` directories and converts Kiro's three-file spec format (`requirements.md`, `design.md`, `tasks.md`) into supersigil MDX documents. This is the migration path from existing Kiro workflows to supersigil's structured, verifiable spec format.

The module is a library crate exposing a public API that the CLI crate (`supersigil-cli`) will consume via the `supersigil import --from kiro` subcommand. It has no dependency on the CLI — all functionality is testable as a library.

Each Kiro spec directory produces up to three uniquely named MDX files: a requirement document, a design document, and a tasks document. The conversion preserves prose, code blocks, and mermaid diagrams while mapping Kiro's structured elements (EARS criteria, `Validates:` references, ordered task lists) into supersigil MDX components (`<Criterion>`, `<Validates>`, `<Task>`, etc.). Ambiguous conversion points are marked with `<!-- TODO(supersigil-import): ... -->` comments for human review.

## Requirements

- **Importer**: The `supersigil-import` crate, responsible for discovering Kiro spec directories and converting their contents into supersigil MDX strings.
- **Kiro_Spec_Dir**: A directory under `.kiro/specs/` containing one or more of `requirements.md`, `design.md`, `tasks.md`. The directory name is the feature name (e.g., `.kiro/specs/parser-and-config/`).
- **Feature_Name**: The name of a Kiro spec directory (e.g., `parser-and-config`), derived from the directory name. Used as the basis for generated document IDs.
- **Import_Plan**: A structured preview of what the Importer would produce: the list of MDX documents to be created, the mappings performed, and any ambiguity markers. Returned by the Importer for `--dry-run` and for programmatic inspection before writing files.
- **Import_Result**: The output of a completed import: the list of generated MDX documents (as in-memory strings with their intended output paths), plus diagnostics.
- **Criterion_ID**: An auto-generated identifier for a `<Criterion>` component, derived from the requirement number and criterion index within that requirement (e.g., `req-1-1` for Requirement 1, criterion 1; `req-1-8a` for Requirement 1, criterion 8a). Criterion indices preserve the source numbering, which may be alphanumeric.
- **EARS_Criterion**: A line in a Kiro `requirements.md` acceptance criteria list that follows EARS notation (e.g., `WHEN...THE System SHALL...`, `THE System SHALL...`, `IF...THEN THE System SHALL...`).
- **Task_ID**: An auto-generated identifier for a `<Task>` component, derived from the task number in the Kiro task list (e.g., `task-1` for top-level task 1, `task-1-1` for sub-task 1.1).
- **Requirement_Ref**: A reference string like `Requirements 1.2` or `Requirements 1.2, 3.4` found in Kiro design.md `Validates:` lines or tasks.md `_Requirements:_` / `**Validates:**` metadata, mapping to specific acceptance criteria. May include alphanumeric indices (e.g., `8a`) and range notation (e.g., `1.1–1.3`).
- **Ambiguity_Marker**: An HTML comment of the form `<!-- TODO(supersigil-import): [description] -->` inserted at conversion points where the mapping is uncertain or unresolvable.
- **MDX_Output**: A string containing a complete supersigil MDX document (front matter + body) ready to be written to a `.mdx` file.
- **ID_Prefix**: An optional string prefix for generated document IDs (e.g., `my-project/` produces `my-project/req/feature-name`). When omitted, no prefix is used and IDs take the form `req/{feature_name}`. See Requirement 16 for the full construction rule.
- **Output_Dir**: The directory where generated MDX files are written. Defaults to `specs/`.

## Requirement 1: Kiro Spec Directory Discovery

As a developer migrating from Kiro, I want the Importer to find all Kiro spec directories automatically, so that I do not have to specify each one manually.

<AcceptanceCriteria>
  <Criterion id="req-1-1">
    WHEN a `.kiro/specs/` directory exists and contains subdirectories, THE Importer SHALL discover each subdirectory as a Kiro_Spec_Dir.
  </Criterion>
  <Criterion id="req-1-2">
    WHEN a Kiro_Spec_Dir contains at least one of `requirements.md`, `design.md`, or `tasks.md`, THE Importer SHALL include that directory in the import.
  </Criterion>
  <Criterion id="req-1-3">
    WHEN a Kiro_Spec_Dir contains none of `requirements.md`, `design.md`, or `tasks.md`, THE Importer SHALL skip that directory and emit a diagnostic warning identifying the skipped directory.
  </Criterion>
  <Criterion id="req-1-4">
    IF the `.kiro/specs/` directory does not exist, THEN THE Importer SHALL return an error indicating the directory was not found.
  </Criterion>
  <Criterion id="req-1-5">
    THE Importer SHALL derive the Feature_Name from the Kiro_Spec_Dir directory name.
  </Criterion>
</AcceptanceCriteria>

## Requirement 2: Requirements.md Parsing

As a developer migrating from Kiro, I want the Importer to parse Kiro requirements.md files, so that my existing acceptance criteria are converted into supersigil `<Criterion>` components.

<AcceptanceCriteria>
  <Criterion id="req-2-1">
    WHEN a Kiro `requirements.md` file is provided, THE Importer SHALL extract the introduction text, glossary, and all requirement sections.
  </Criterion>
  <Criterion id="req-2-2">
    WHEN a requirement section contains a `**User Story:**` line, THE Importer SHALL extract the user story text.
  </Criterion>
  <Criterion id="req-2-3">
    WHEN a requirement section contains an `#### Acceptance Criteria` subsection, THE Importer SHALL extract each numbered criterion line as an EARS_Criterion.
  </Criterion>
  <Criterion id="req-2-4">
    WHEN a requirement section has a heading matching `### Requirement N: Title`, THE Importer SHALL extract the requirement number `N` and the title.
  </Criterion>
  <Criterion id="req-2-5">
    IF a `requirements.md` file contains no parseable requirement sections, THEN THE Importer SHALL emit a diagnostic warning and produce an MDX document containing only the raw prose.
  </Criterion>
  <Criterion id="req-2-6">
    WHEN the Kiro requirements.md contains a heading matching `# Requirements Document` or `# Requirements Document: Title`, THE Importer SHALL extract the title (if present) for use in the MDX front matter `title` field. WHEN no title is present in the heading, THE Importer SHALL use the Feature_Name as the title.
  </Criterion>
</AcceptanceCriteria>

## Requirement 3: Criterion ID Generation

As a developer migrating from Kiro, I want each acceptance criterion to receive a stable, predictable ID, so that design and task documents can reference them.

<AcceptanceCriteria>
  <Criterion id="req-3-1">
    THE Importer SHALL generate a Criterion_ID for each EARS_Criterion using the pattern `req-{requirement_number}-{criterion_index}`, where `criterion_index` is the source numbering from the Kiro acceptance criteria list (e.g., `1`, `2`, `8a`, `8b`). WHEN the source numbering uses alphanumeric identifiers (e.g., `8a`), the Criterion_ID SHALL preserve the alphanumeric form (e.g., `req-1-8a`).
  </Criterion>
  <Criterion id="req-3-2">
    THE Importer SHALL ensure all generated Criterion_IDs are unique within a single requirements MDX document.
  </Criterion>
  <Criterion id="req-3-3">
    WHEN a Criterion_ID would collide with another within the same document, THE Importer SHALL append a disambiguating suffix and emit an Ambiguity_Marker.
  </Criterion>
</AcceptanceCriteria>

## Requirement 4: Requirements MDX Emission

As a developer migrating from Kiro, I want the Importer to produce a valid supersigil MDX requirement document from my Kiro requirements.md, so that my criteria are immediately usable in the supersigil verification graph.

<AcceptanceCriteria>
  <Criterion id="req-4-1">
    THE Importer SHALL produce an MDX_Output with front matter containing `supersigil.id` set to `{id_prefix}/req/{feature_name}`, `supersigil.type` set to `requirement`, and `supersigil.status` set to `draft`.
  </Criterion>
  <Criterion id="req-4-2">
    THE Importer SHALL emit the introduction text as prose at the top of the MDX body, before any requirement sections.
  </Criterion>
  <Criterion id="req-4-3">
    FOR EACH requirement section, THE Importer SHALL emit the user story text as prose followed by an `<AcceptanceCriteria>` block containing one `<Criterion>` per EARS_Criterion, each with its generated Criterion_ID as the `id` attribute and the criterion text as the body.
  </Criterion>
  <Criterion id="req-4-4">
    THE Importer SHALL emit one `<AcceptanceCriteria>` block per requirement section (not a single global block), preserving the per-requirement grouping from the Kiro source.
  </Criterion>
  <Criterion id="req-4-5">
    WHEN the Kiro requirements.md contains a glossary section, THE Importer SHALL preserve the glossary as prose after the introduction and before the first requirement section in the MDX output.
  </Criterion>
</AcceptanceCriteria>

## Requirement 5: Design.md Parsing

As a developer migrating from Kiro, I want the Importer to parse Kiro design.md files, so that my design prose, diagrams, and property-to-requirement mappings are preserved.

<AcceptanceCriteria>
  <Criterion id="req-5-1">
    WHEN a Kiro `design.md` file is provided, THE Importer SHALL extract all sections including overview, architecture, components, data models, correctness properties, error handling, and testing strategy.
  </Criterion>
  <Criterion id="req-5-2">
    WHEN a section contains a mermaid code block (` ```mermaid `), THE Importer SHALL preserve the code block verbatim in the MDX output.
  </Criterion>
  <Criterion id="req-5-3">
    WHEN a section contains a Rust or other language code block, THE Importer SHALL preserve the code block verbatim in the MDX output.
  </Criterion>
  <Criterion id="req-5-4">
    WHEN a correctness property section contains a `**Validates: Requirements X.Y, Z.W**` line, THE Importer SHALL extract the Requirement_Ref values (`X.Y`, `Z.W`). WHEN a `**Validates:**` line references a non-requirement target (e.g., `Design Decision 5`), THE Importer SHALL preserve the line as prose and emit an Ambiguity_Marker noting the non-requirement target.
  </Criterion>
  <Criterion id="req-5-5">
    WHEN a design.md contains a heading matching `# Design Document: Title` or `# Design: Title`, THE Importer SHALL extract the title. WHEN no matching heading is found, THE Importer SHALL use the Feature_Name as the title.
  </Criterion>
</AcceptanceCriteria>

## Requirement 6: Validates Reference Resolution

As a developer migrating from Kiro, I want `Validates: Requirements X.Y` references in my design.md to be converted into `<Validates>` components with correct criterion refs, so that the verification chain is preserved.

<AcceptanceCriteria>
  <Criterion id="req-6-1">
    WHEN a Requirement_Ref `X.Y` can be mapped to a Criterion_ID in the same feature's requirements document (where `X` is the requirement number and `Y` is the criterion index), THE Importer SHALL emit a `<Validates>` component with `refs` pointing to `{id_prefix}/req/{feature_name}#req-{X}-{Y}`.
  </Criterion>
  <Criterion id="req-6-2">
    WHEN a Requirement_Ref `X.Y` cannot be unambiguously resolved to a Criterion_ID (e.g., the requirement number or criterion index does not exist in the parsed requirements), THE Importer SHALL emit an Ambiguity_Marker at the conversion point describing the unresolvable reference.
  </Criterion>
  <Criterion id="req-6-3">
    WHEN multiple Requirement_Refs appear in a single `Validates:` line, THE Importer SHALL combine all resolvable refs into a single `<Validates>` component with a comma-separated `refs` attribute. WHEN some refs in the line resolve and others do not, THE Importer SHALL emit the `<Validates>` component containing only the resolvable refs AND emit an Ambiguity_Marker for each unresolvable ref.
  </Criterion>
</AcceptanceCriteria>

## Requirement 7: Design MDX Emission

As a developer migrating from Kiro, I want the Importer to produce a valid supersigil MDX design document from my Kiro design.md, so that my technical design is linked to my requirements.

<AcceptanceCriteria>
  <Criterion id="req-7-1">
    THE Importer SHALL produce an MDX_Output with front matter containing `supersigil.id` set to `{id_prefix}/design/{feature_name}`, `supersigil.type` set to `design`, and `supersigil.status` set to `draft`.
  </Criterion>
  <Criterion id="req-7-2">
    WHEN the same Kiro_Spec_Dir contains a `requirements.md` file, THE Importer SHALL emit an `<Implements>` component with `refs` pointing to `{id_prefix}/req/{feature_name}`. WHEN no `requirements.md` exists for the feature, THE Importer SHALL omit the `<Implements>` component and emit an Ambiguity_Marker noting the missing requirements document.
  </Criterion>
  <Criterion id="req-7-3">
    THE Importer SHALL preserve all prose, code blocks, and mermaid diagrams from the Kiro design.md in the MDX body.
  </Criterion>
  <Criterion id="req-7-4">
    WHEN a correctness property has resolved `Validates:` references, THE Importer SHALL emit a `<Validates>` component after the property's prose.
  </Criterion>
  <Criterion id="req-7-5">
    WHEN a correctness property has only unresolvable `Validates:` references, THE Importer SHALL emit an Ambiguity_Marker instead of a `<Validates>` component. WHEN a correctness property has a mix of resolvable and unresolvable references, THE Importer SHALL emit a `<Validates>` component with the resolvable refs and an Ambiguity_Marker for each unresolvable ref.
  </Criterion>
</AcceptanceCriteria>

## Requirement 8: Tasks.md Parsing

As a developer migrating from Kiro, I want the Importer to parse Kiro tasks.md files, so that my implementation plan is converted into supersigil `<Task>` components.

<AcceptanceCriteria>
  <Criterion id="req-8-1">
    WHEN a Kiro `tasks.md` file is provided, THE Importer SHALL extract all top-level tasks from the ordered task list.
  </Criterion>
  <Criterion id="req-8-2">
    WHEN a top-level task has sub-tasks (lines indented under the parent with a task marker like `- [x] 1.1 Sub-task title`), THE Importer SHALL extract each sub-task as a child of the parent task.
  </Criterion>
  <Criterion id="req-8-3">
    WHEN a task line matches the pattern `- [marker] N. Task title` or `- [marker] N.M Sub-task title`, THE Importer SHALL extract the task number, optional sub-task number, title, and status marker.
  </Criterion>
  <Criterion id="req-8-4">
    THE Importer SHALL map Kiro task status markers to supersigil task statuses: `[x]` to `done`, `[ ]` to `ready`, `[-]` to `in-progress`, `[~]` to `draft` (`[~]` is Kiro's "queued" status, mapped to `draft` as the closest supersigil equivalent).
  </Criterion>
  <Criterion id="req-8-5">
    WHEN a task or sub-task has description lines (indented lines following the task line that are not sub-tasks), THE Importer SHALL extract the description text.
  </Criterion>
  <Criterion id="req-8-6">
    WHEN a task or sub-task has a metadata line matching `_Requirements: X.Y, Z.W_`, `_Validates: Requirements X.Y, Z.W_`, `**Validates: Requirements X.Y, Z.W**`, or `**Requirements: X.Y, Z.W**`, THE Importer SHALL extract the Requirement_Ref values. Both italic (`_..._`) and bold (`**...**`) markup forms are recognized.
  </Criterion>
  <Criterion id="req-8-7">
    WHEN a task or sub-task has a metadata line matching `_Requirements: N/A_`, `_Requirements: N/A (...)_`, or any `_Requirements: ..._` / `**Requirements: ...**` value that does not parse as a ref list (e.g., `_Requirements: (test infrastructure)_`), THE Importer SHALL treat it as having no requirement references (no `implements` attribute, no Ambiguity_Marker). The non-ref value SHALL be preserved as a comment in the `<Task>` body.
  </Criterion>
  <Criterion id="req-8-8">
    WHEN the Kiro tasks.md contains a heading matching `# Implementation Plan: Title` or `# Tasks: Title`, THE Importer SHALL extract the title for use in the MDX front matter `title` field. WHEN no matching heading is found, THE Importer SHALL use the Feature_Name as the title.
  </Criterion>
</AcceptanceCriteria>

## Requirement 9: Task ID Generation

As a developer migrating from Kiro, I want each task to receive a stable, predictable ID, so that task dependencies and references are consistent.

<AcceptanceCriteria>
  <Criterion id="req-9-1">
    THE Importer SHALL generate a Task_ID for each top-level task using the pattern `task-{N}` where `N` is the task number from the Kiro task list.
  </Criterion>
  <Criterion id="req-9-2">
    THE Importer SHALL generate a Task_ID for each sub-task using the pattern `task-{N}-{M}` where `N` is the parent task number and `M` is the sub-task number.
  </Criterion>
  <Criterion id="req-9-3">
    THE Importer SHALL ensure all generated Task_IDs are unique within a single tasks MDX document.
  </Criterion>
  <Criterion id="req-9-4">
    WHEN a Task_ID would collide with another within the same document (e.g., duplicate task numbers in the source), THE Importer SHALL append a disambiguating suffix and emit an Ambiguity_Marker.
  </Criterion>
</AcceptanceCriteria>

## Requirement 10: Task Dependency Generation

As a developer migrating from Kiro, I want sequential tasks to have `depends` attributes, so that the implementation order is preserved in the supersigil format.

<AcceptanceCriteria>
  <Criterion id="req-10-1">
    WHEN a top-level task has a preceding sibling top-level task, THE Importer SHALL set the `depends` attribute to the preceding sibling's Task_ID.
  </Criterion>
  <Criterion id="req-10-2">
    WHEN a sub-task has a preceding sibling sub-task within the same parent, THE Importer SHALL set the `depends` attribute to the preceding sibling's Task_ID.
  </Criterion>
  <Criterion id="req-10-3">
    WHEN a task is the first in its sibling group (first top-level task or first sub-task within a parent), THE Importer SHALL omit the `depends` attribute.
  </Criterion>
</AcceptanceCriteria>

## Requirement 11: Task Implements Resolution

As a developer migrating from Kiro, I want `_Requirements: X.Y_` metadata on tasks to be converted into `implements` attributes on `<Task>` components, so that task-to-criterion traceability is preserved.

<AcceptanceCriteria>
  <Criterion id="req-11-1">
    WHEN a task's Requirement_Ref `X.Y` can be mapped to a Criterion_ID in the same feature's requirements document, THE Importer SHALL set the `implements` attribute on the `<Task>` component to `{id_prefix}/req/{feature_name}#req-{X}-{Y}`.
  </Criterion>
  <Criterion id="req-11-2">
    WHEN a task has multiple resolvable Requirement_Refs, THE Importer SHALL combine them into a comma-separated `implements` attribute.
  </Criterion>
  <Criterion id="req-11-3">
    WHEN a task's Requirement_Ref cannot be resolved, THE Importer SHALL emit an Ambiguity_Marker inside the `<Task>` body describing the unresolvable reference and omit that ref from the `implements` attribute.
  </Criterion>
</AcceptanceCriteria>

## Requirement 12: Tasks MDX Emission

As a developer migrating from Kiro, I want the Importer to produce a valid supersigil MDX tasks document from my Kiro tasks.md, so that my implementation plan is usable in the supersigil graph.

<AcceptanceCriteria>
  <Criterion id="req-12-1">
    THE Importer SHALL produce an MDX_Output with front matter containing `supersigil.id` set to `{id_prefix}/tasks/{feature_name}`, `supersigil.type` set to `tasks`, and `supersigil.status` set to `draft`.
  </Criterion>
  <Criterion id="req-12-2">
    THE Importer SHALL emit each top-level task as a `<Task>` component with `id`, `status`, and optional `depends` and `implements` attributes.

  <Criterion id="req-12-3">
    WHEN a top-level task has sub-tasks, THE Importer SHALL emit the sub-tasks as nested `<Task>` components within the parent `<Task>`.
  </Criterion>
  <Criterion id="req-12-4">
    THE Importer SHALL include the task description text as the body of each `<Task>` component.
  </Criterion>
  <Criterion id="req-12-5">
    WHEN the Kiro tasks.md contains overview or notes sections, THE Importer SHALL preserve them as prose in the MDX output.
  </Criterion>
</AcceptanceCriteria>

## Requirement 13: Ambiguity Marker Insertion

As a developer migrating from Kiro, I want uncertain conversion points to be clearly marked, so that I can review and fix them after import.

<AcceptanceCriteria>
  <Criterion id="req-13-1">
    WHEN a Requirement_Ref cannot be resolved to a Criterion_ID, THE Importer SHALL insert an Ambiguity_Marker of the form `<!-- TODO(supersigil-import): Could not resolve reference 'Requirements X.Y' to a criterion ID -->`.

## Requirement 15: MDX File Writing

As a developer migrating from Kiro, I want the Importer to write the generated MDX files to disk, so that I can immediately use them with supersigil.

<AcceptanceCriteria>
  <Criterion id="req-15-1">
    WHEN the Importer is invoked in write mode, THE Importer SHALL write each generated MDX_Output to a file at `{output_dir}/{feature_name}/{feature_name}.{type}.mdx` where `type` is `req`, `design`, or `tasks`.
  </Criterion>
  <Criterion id="req-15-2">
    WHEN the output directory does not exist, THE Importer SHALL create it (including parent directories).
  </Criterion>
  <Criterion id="req-15-3">
    IF a file already exists at the target path and the `force` option is not set, THEN THE Importer SHALL return an error identifying the conflicting path rather than overwriting the file. WHEN the `force` option is set, THE Importer SHALL overwrite existing files.
  </Criterion>
  <Criterion id="req-15-4">
    THE Importer SHALL use `specs/` as the default Output_Dir when no output directory is specified.
  </Criterion>
  <Criterion id="req-15-5">
    WHEN writing multiple files for a multi-feature import, THE Importer SHALL use best-effort semantics: files are written sequentially, and a failure on a later file does not roll back previously written files. The `force` option (Req 15.3) enables re-running the import to recover from partial writes.
  </Criterion>
</AcceptanceCriteria>

## Requirement 16: Document ID Construction

As a developer migrating from Kiro, I want to specify a prefix for generated document IDs, so that imported specs fit my project's ID naming convention.

<AcceptanceCriteria>
  <Criterion id="req-16-1">
    WHEN an ID_Prefix is provided, THE Importer SHALL construct document IDs as `{id_prefix}/{type_hint}/{feature_name}` (e.g., `my-project/parser-and-config/req`). The ID_Prefix MUST NOT have a trailing slash; the Importer SHALL strip any trailing slash from the provided prefix before use.
  </Criterion>
  <Criterion id="req-16-2">
    WHEN no ID_Prefix is provided, THE Importer SHALL construct document IDs as `{type_hint}/{feature_name}` (e.g., `parser-and-config/req`) with no leading slash.
  </Criterion>
  <Criterion id="req-16-3">
    THE `type_hint` segment SHALL be `req` for requirements documents, `design` for design documents, and `tasks` for tasks documents.
  </Criterion>
  <Criterion id="req-16-4">
    All requirements that reference "the document ID" (Reqs 4.1, 6.1, 7.1, 7.2, 11.1, 12.1) SHALL use the ID as constructed by this requirement. The `{id_prefix}/...` notation in those requirements is shorthand for the conditional construction defined here.
  </Criterion>
</AcceptanceCriteria>

## Requirement 20: Requirement Ref Parsing

As a developer migrating from Kiro, I want the Importer to correctly parse all forms of requirement references found in Kiro specs, so that cross-references are resolved accurately.

This requirement applies only to strings that have been identified as requirement reference values by the calling context (Reqs 5.4, 8.6). Sentinel values like `N/A` and non-ref annotations are handled before ref parsing is invoked (see Req 8.7) and are never passed to the ref parser.

<AcceptanceCriteria>
  <Criterion id="req-20-1">
    WHEN a reference string matches the pattern `Requirements X.Y` (single reference), THE Importer SHALL extract requirement number `X` and criterion index `Y`. Both `X` and `Y` may be alphanumeric (e.g., `8a`, `8b`).
  </Criterion>
  <Criterion id="req-20-2">
    WHEN a reference string matches the pattern `Requirements X.Y, Z.W` (comma-separated list), THE Importer SHALL extract all `(requirement_number, criterion_index)` pairs.
  </Criterion>
  <Criterion id="req-20-3">
    WHEN a reference string matches the pattern `X.Y, Z.W` (without the `Requirements` prefix), THE Importer SHALL extract all pairs.
  </Criterion>
  <Criterion id="req-20-4">
    WHEN a reference string matches the pattern `Requirements X.Y, X.Z` (multiple criteria from the same requirement), THE Importer SHALL extract each pair individually.
  </Criterion>
  <Criterion id="req-20-5">
    WHEN a reference string contains a range pattern `X.Y–X.Z` or `X.Y-X.Z` (en-dash or hyphen between two refs with the same requirement number and numeric criterion indices), THE Importer SHALL expand the range into individual refs `X.Y, X.(Y+1), ..., X.Z`. WHEN the criterion indices in a range are non-numeric (e.g., `8a–8c`), THE Importer SHALL emit an Ambiguity_Marker for the range and preserve the original text.
  </Criterion>
  <Criterion id="req-20-6">
    WHEN a reference string contains values that do not match the `X.Y` pattern (e.g., bare `X` without a dot), THE Importer SHALL emit an Ambiguity_Marker for the unparseable portion.
  </Criterion>
</AcceptanceCriteria>

## Requirement 21: MDX Front Matter Formatting

As a developer migrating from Kiro, I want the generated MDX front matter to be valid YAML that supersigil can parse, so that imported documents work immediately with `supersigil verify`.

<AcceptanceCriteria>
  <Criterion id="req-21-1">
    THE Importer SHALL emit front matter delimited by `---` on its own line, with the `supersigil:` namespace containing `id`, `type`, and `status` fields.
  </Criterion>
  <Criterion id="req-21-2">
    THE Importer SHALL emit the `title` field outside the `supersigil:` namespace, set to the extracted document title.
  </Criterion>
  <Criterion id="req-21-3">
    FOR ALL generated MDX documents, parsing the front matter with the supersigil parser SHALL produce a valid `Frontmatter` with the expected `id`, `doc_type`, and `status` values. This is the round-trip property for the import pipeline.
  </Criterion>
</AcceptanceCriteria>

## Requirement 22: Task Optional Marker Handling

As a developer migrating from Kiro, I want optional tasks (marked with `*` in Kiro) to be preserved in the import, so that no task metadata is lost.

<AcceptanceCriteria>
  <Criterion id="req-22-1">
    WHEN a Kiro task line contains an optional marker (`*` after the status bracket, e.g., `- [ ]* 2.1 Optional task`), THE Importer SHALL include the task in the output and insert an Ambiguity_Marker noting the optional status, since supersigil `<Task>` has no `optional` attribute.
  </Criterion>
</AcceptanceCriteria>
