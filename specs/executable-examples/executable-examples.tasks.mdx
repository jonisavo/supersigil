## Overview

Implementation follows the dependency order from the design: core types
first, then parser and verify infrastructure, then the matching engine
and runner dispatch, then evidence integration and CLI orchestration.
Each task is TDD: tests before implementation, with `cargo nextest run`
green before moving on.

<Task id="task-1" status="done" implements="executable-examples/req#req-1-1, executable-examples/req#req-1-2, executable-examples/req#req-1-3, executable-examples/req#req-1-5">
  Add core types and component definitions. Register `Example` and
  `Expected` as built-in components in `component_defs.rs` with the
  attribute sets, referenceable/verifiable flags, and `references` edge
  semantics from the design, including optional `lang` derivation from
  the first fenced code block. Add `CodeBlock` (with `content_offset`)
  to `types.rs`. Add `EvidenceKind::Example` to `supersigil-evidence`.
</Task>

<Task id="task-2" status="done" implements="executable-examples/req#req-6-1, executable-examples/req#req-6-2, executable-examples/req#req-6-3">
  Add the `[examples]` configuration model. Add `ExamplesConfig` and
  `RunnerConfig` to the config model in `supersigil-core`, with
  deserialization from `supersigil.toml`. Include `timeout`, `parallelism`,
  and `runners` map. Validate placeholder syntax (`{file}`, `{dir}`,
  `{lang}`, `{name}`) in runner command templates.
</Task>

<Task id="task-3" status="done" depends="task-1" implements="executable-examples/req#req-1-8">
  Amend the parser to extract code blocks. Extend the extractor in
  `supersigil-parser` to populate `code_blocks: Vec<CodeBlock>` on
  `ExtractedComponent` for `Example` and `Expected` components, including
  `content_offset` relative to the normalized source. Other components
  get empty vectors.
</Task>

<Task id="task-4" status="done" implements="executable-examples/req#req-4-1">
  Split `verify()` into composable parts. Extract `verify_structural()`
  (all rules except coverage and hooks) and `verify_coverage()` (coverage
  rules only) as public functions in `supersigil-verify`. Keep the
  existing `verify()` as a convenience that calls both plus hooks.
</Task>

<Task id="task-5" status="done" depends="task-1, task-3, task-4" implements="executable-examples/req#req-1-2">
  Add structural lint rules for Example and Expected. Add a rule that
  `Expected` must be a direct child of `Example` (analogous to
  `check_verified_by_placement`). Add code block cardinality validation:
  `Example` must have exactly one code block, `Expected` at most one.
  Add `env` attribute format validation (each item must contain `=`).
</Task>

<Task id="task-6" status="done" implements="executable-examples/req#req-1-6, executable-examples/req#req-1-7, executable-examples/req#req-3-1, executable-examples/req#req-3-2, executable-examples/req#req-3-3, executable-examples/req#req-3-5, executable-examples/req#req-3-6">
  Implement the output matching engine. Build the matcher in
  `supersigil-verify/src/examples/matcher.rs`: text (trimmed exact),
  JSON (deep compare with `<any-string>`, `<any-number>`, `<any-uuid>`,
  `<any-iso8601>` wildcards and path-aware diffs), regex (full match),
  status check, contains check, and conjunctive check logic. Include
  property tests for JSON wildcard edge cases.
</Task>

<Task id="task-7" status="done" depends="task-2" implements="executable-examples/req#req-2-1, executable-examples/req#req-2-2, executable-examples/req#req-2-3, executable-examples/req#req-2-4, executable-examples/req#req-2-5, executable-examples/req#req-2-6">
  Implement runner dispatch and execution. Build runner resolution
  (config takes precedence over built-ins), placeholder interpolation,
  and the three built-in runners: `sh` (subprocess), `cargo-test`

  ureq). Implement setup script execution, env variable injection, and
  timeout enforcement. Integration tests using the `sh` runner against
  real subprocess execution.
</Task>

<Task id="task-8" status="done" depends="task-6, task-7" implements="executable-examples/req#req-1-4, executable-examples/req#req-4-2, executable-examples/req#req-4-3">

  executor that orchestrates runner dispatch and output matching. On
  pass with `verifies`, produce `VerificationEvidenceRecord` with
  `EvidenceKind::Example`. On failure, produce findings with example id,
  runner, actual vs expected diff, and targeted criteria. Merge example
  evidence into the `ArtifactGraph`.
</Task>
