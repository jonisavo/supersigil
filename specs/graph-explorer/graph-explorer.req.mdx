## Introduction

A browser-based interactive visualization of the spec graph. Developers and
stakeholders can explore how specs relate, drill into document internals, trace
impact of changes, and filter by type or status — without reading individual MDX
files.

The feature has two delivery surfaces: an Astro docs page at `/explore/` and a
standalone `supersigil explore` CLI command that generates a self-contained HTML
file. Both share the same visualization module.

## Definitions

- **Graph_JSON**: A JSON serialization of the document graph containing
  documents, their components, and document-level edges. Produced by
  `supersigil graph --format json`.
- **Explorer_View**: The interactive browser-based visualization rendered from
  Graph_JSON. Consists of a force-directed graph canvas, filter bar, detail
  sidebar, and search.
- **Document_Cluster**: A visual grouping of a single document's node in the
  graph. Expandable to reveal internal components.
- **Impact_Trace**: A transitive highlight of all documents downstream from a
  selected node via DependsOn, Implements, or References edges.

## Requirement 1: JSON Graph Export

As a developer or tool author, I want to export the document graph as
structured JSON, so that it can be consumed by the visual explorer or other
downstream tools.

<AcceptanceCriteria>
  <Criterion id="req-1-1">
    THE `graph` command SHALL accept `--format json` in addition to
    existing `mermaid` and `dot` formats.
  </Criterion>
  <Criterion id="req-1-2">
    Graph_JSON SHALL contain a `documents` array where each entry includes
    `id`, `doc_type` (raw frontmatter value or null), `status` (or null),
    `title` (from frontmatter extra, falling back to document ID), and a
    `components` array.
  </Criterion>
  <Criterion id="req-1-3">
    Each component in Graph_JSON SHALL include `id` (from component
    attributes, null if absent), `kind` (component name), and `body` (body
    text). Decision components SHALL include nested `children` for
    Rationale and Alternative sub-components.
  </Criterion>
  <Criterion id="req-1-4">
    Graph_JSON SHALL contain an `edges` array with `from`, `to`, and `kind`
    fields. Edges SHALL be sourced from the DocumentGraph reverse mappings,
    not from iterating top-level components only.
  </Criterion>
  <Criterion id="req-1-5">
    Graph_JSON SHALL be written to stdout. Summary output SHALL go to stderr,
    consistent with the existing mermaid and dot formats.
  </Criterion>
</AcceptanceCriteria>

## Requirement 2: Standalone Explorer Command

As a developer, I want to run `supersigil explore` and immediately see an
interactive graph in my browser, so that I can orient myself in the spec
landscape without configuring anything.

<AcceptanceCriteria>
  <Criterion id="req-2-1">
    THE `explore` command SHALL generate a self-contained HTML file embedding
    the visualization JS module and the graph JSON data inline.
  </Criterion>
  <Criterion id="req-2-2">
    THE `explore` command SHALL open the generated HTML in the system default
    browser. An `--output` flag SHALL allow writing to a specified path
    instead.
  </Criterion>
</AcceptanceCriteria>

## Requirement 3: Astro Docs Page

As a team member viewing the documentation site, I want an `/explore/` page
showing the interactive graph, so that the spec landscape is browsable
alongside documentation.

<AcceptanceCriteria>
  <Criterion id="req-3-1">
    THE Astro site SHALL include a page at `/explore/` hosting the same
    visualization module used by the standalone explorer.
  </Criterion>
  <Criterion id="req-3-2">
    Graph data SHALL be generated at build time via a prebuild script running
    `supersigil graph --format json` and writing to a static JSON file.
  </Criterion>
</AcceptanceCriteria>

## Requirement 4: Graph Visualization

As a developer, I want to see the spec graph as an interactive, force-directed
visualization with clustered document nodes, so that I can understand
relationships spatially.

<AcceptanceCriteria>
  <Criterion id="req-4-1">
    THE Explorer_View SHALL render documents as nodes in a clustered
    force-directed layout. Nodes SHALL be visually grouped into
    Document_Clusters with dashed borders.
  </Criterion>
  <Criterion id="req-4-2">
    Document-level edges (Implements, DependsOn, References) SHALL be
    rendered as lines between document nodes, with edge kind as the label.
  </Criterion>
  <Criterion id="req-4-3">
    THE Explorer_View SHALL support pan (drag background), zoom (scroll),
    and drag (reposition individual nodes) interactions.
  </Criterion>
  <Criterion id="req-4-4">
    Double-clicking a Document_Cluster SHALL expand it to reveal its
    internal components (Criteria, Tasks, Decisions with Alternatives and
    Rationale). Double-clicking again SHALL collapse it.
  </Criterion>
</AcceptanceCriteria>

## Requirement 5: Detail Sidebar

As a developer, I want to click a document node and see its details, so that
I can inspect type, status, edges, and components without leaving the graph.

<AcceptanceCriteria>
  <Criterion id="req-5-1">
    Clicking a document node SHALL open a detail sidebar showing: document
    ID, doc type badge, status badge, list of edges (incoming and outgoing),
    and list of components with kind-colored indicators.
  </Criterion>
  <Criterion id="req-5-2">
    The selected node SHALL receive a gold highlight ring. Clicking the same
    node or pressing Escape SHALL close the sidebar.
  </Criterion>
</AcceptanceCriteria>

## Requirement 6: Filtering

As a developer, I want to filter the graph by document type and status, so
that I can focus on the specs relevant to my current work.

<AcceptanceCriteria>
  <Criterion id="req-6-1">
    THE Explorer_View SHALL provide toggle chips for document types (at
    minimum: requirements, design, adr, tasks). Active filters SHALL use
    gold styling; inactive SHALL be dimmed.
  </Criterion>
  <Criterion id="req-6-2">
    THE Explorer_View SHALL provide a status filter (all, draft, approved,
    implemented, deprecated, or other statuses present in the data).
  </Criterion>
  <Criterion id="req-6-3">
    Filtered-out nodes SHALL fade to near-invisible rather than being removed,
    maintaining spatial context for visible nodes.
  </Criterion>
</AcceptanceCriteria>

## Requirement 7: Impact Trace

As a developer, I want to trace the downstream impact of a spec change, so
that I can understand what would be affected before modifying a document.

<AcceptanceCriteria>
  <Criterion id="req-7-1">
    THE Explorer_View SHALL provide an impact trace action (via detail panel
    button or right-click context menu) that highlights all documents
    transitively downstream from the selected node.
  </Criterion>
  <Criterion id="req-7-2">
    During an active Impact_Trace, the traced subgraph SHALL render at full
    opacity with gold edge highlights. All other nodes and edges SHALL dim.
    Clicking the background SHALL clear the trace.
  </Criterion>
</AcceptanceCriteria>

## Requirement 8: Search

As a developer, I want to search for a specific document by name and jump to
it in the graph, so that I can quickly navigate large spec graphs.

<AcceptanceCriteria>
  <Criterion id="req-8-1">
    THE Explorer_View SHALL include a search input that filters nodes by
    fuzzy match on document ID and title as the user types.
  </Criterion>
  <Criterion id="req-8-2">
    Selecting a search result SHALL zoom to the matching node and open its
    detail sidebar. The `/` key SHALL focus the search input.
  </Criterion>
</AcceptanceCriteria>

## Requirement 9: URL Deep-Linking

As a developer, I want to share a link to a specific graph view, so that
teammates can see the same node selection, trace, or filter state.

<AcceptanceCriteria>
  <Criterion id="req-9-1">
    THE Explorer_View SHALL use hash-based routing to encode the current
    view state. At minimum: `#/doc/{id}` for node selection,
    `#/doc/{id}/trace` for active impact trace, and `#/filter/{params}`
    for filter state.
  </Criterion>
  <Criterion id="req-9-2">
    THE URL hash SHALL update on interaction so views are
    shareable and bookmarkable. Loading a URL with a hash SHALL restore
    the encoded view state.
  </Criterion>
</AcceptanceCriteria>

## Requirement 10: Visual Design Consistency

As a user of the supersigil docs site, I want the graph explorer to feel
like a native part of the site, so that the experience is cohesive.

<AcceptanceCriteria>
  <Criterion id="req-10-1">
    THE Explorer_View SHALL use the existing Obsidian Glow design tokens
    (CSS custom properties from landing-tokens.css) for all colors,
    typography, and spacing. It SHALL support both dark and light themes
    via Starlight's theme toggle.
  </Criterion>
  <Criterion id="req-10-2">
    Node stroke colors SHALL differentiate document types: teal for
    requirements, green for design, gold for ADR. Node size SHALL scale
    with component count.
  </Criterion>
</AcceptanceCriteria>
