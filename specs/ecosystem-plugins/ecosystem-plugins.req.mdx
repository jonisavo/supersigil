## Introduction

This spec now captures only the cross-cutting ecosystem plugin layer that
spans config, CLI orchestration, verification, and reporting.

The crate-local behaviors are recovered separately in:

- `evidence-contract/*` under `crates/supersigil-evidence/specs/`
- `rust-plugin/*` under `crates/supersigil-rust/specs/`
- `verifies-macro/*` under `crates/supersigil-rust-macros/specs/`

## Definitions

- **Enabled_Plugin_Set**: The built-in plugin names resolved from
  `ecosystem.plugins`.
- **Plugin_Evidence_Pipeline**: The current cross-cutting flow that assembles
  plugins, resolves shared test files, lets enabled plugins augment their own
  discovery inputs, extracts explicit evidence, and builds an `ArtifactGraph`.
- **Plugin_Finding**: A verification finding surfaced when plugin discovery
  yields a recoverable diagnostic, fails, or otherwise yields an ecosystem-
  level problem.

## Requirement 1: Ecosystem Activation and Policy

As a workspace maintainer, I want ecosystem behavior configured explicitly, so
that plugin-backed evidence can be enabled, disabled, and scoped predictably.

<AcceptanceCriteria>
  <Criterion id="req-1-1">
    THE config model SHALL expose `ecosystem.plugins` as the Enabled_Plugin_Set
    and SHALL default that set to `["rust"]`.
  </Criterion>
  <Criterion id="req-1-2">
    IF config explicitly sets `plugins = []`, THEN the Plugin_Evidence_Pipeline
    SHALL disable plugin-derived evidence while leaving explicit `VerifiedBy`
    extraction available.
  </Criterion>
  <Criterion id="req-1-3">
    IF config names an unknown built-in plugin, THEN config loading SHALL fail
    before CLI plugin assembly runs.
  </Criterion>
  <Criterion id="req-1-4">
    THE config model SHALL expose the current Rust-specific ecosystem policy
    surface under `ecosystem.rust`, including `validation` and
    `project_scope`.
    <VerifiedBy strategy="file-glob" paths="crates/supersigil-core/tests/config_unit_tests.rs" />
  </Criterion>
</AcceptanceCriteria>

## Requirement 2: Cross-Cutting Evidence Orchestration

As an operator, I want plugin and authored evidence to flow through one
pipeline, so that verification and query surfaces see one merged evidence view.

<AcceptanceCriteria>
  <Criterion id="req-2-1">
    THE CLI plugin layer SHALL assemble built-in plugin instances from the
    Enabled_Plugin_Set and SHALL currently instantiate the Rust plugin for the
    built-in `"rust"` identifier.
  </Criterion>
  <Criterion id="req-2-2">
    THE Plugin_Evidence_Pipeline SHALL resolve shared test files once for
    explicit-evidence extraction and plugin discovery, and SHALL let enabled
    plugins broaden that baseline with plugin-owned discovery inputs.
  </Criterion>
  <Criterion id="req-2-3">
    THE Plugin_Evidence_Pipeline SHALL merge explicit `VerifiedBy` evidence and
    plugin-derived evidence into one `ArtifactGraph` instead of keeping
    separate evidence channels.
  </Criterion>
  <Criterion id="req-2-4">
    IF plugin discovery emits recoverable diagnostics or fails for one enabled
    plugin, THEN the pipeline SHALL surface Plugin_Finding entries and SHALL
    continue returning a verification result instead of aborting the whole
    verification flow.
  </Criterion>
</AcceptanceCriteria>

## Requirement 3: Evidence Surfacing

As a developer inspecting verification results, I want ecosystem-backed
evidence surfaced clearly, so that plugin-derived coverage is visible and
debuggable.

<AcceptanceCriteria>
  <Criterion id="req-3-1">
    THE verification report surface SHALL be able to include an evidence
    summary that shows evidence kind, discovered test identity, source
    location, and provenance.
  </Criterion>
  <Criterion id="req-3-2">
    THE reporting layer SHALL preserve merged provenance strings when multiple
    sources back the same effective evidence record.
  </Criterion>
  <Criterion id="req-3-3">
    THE ecosystem layer SHALL surface plugin discovery diagnostics and
    failures through the existing verification/reporting surfaces rather than
    as configuration or graph-construction failures.
  </Criterion>
</AcceptanceCriteria>

## Requirement 4: Modular Extension Boundary

As a maintainer, I want ecosystem support split cleanly from the document-only
core, so that new ecosystems can be added without changing the authored spec
format.

<AcceptanceCriteria>
  <Criterion id="req-4-1">
    THE language-agnostic `DocumentGraph` and core config model SHALL remain
    usable without taking a compile-time dependency on Rust parsing or proc-
    macro code, so that core tooling stays ecosystem-agnostic.
    <VerifiedBy strategy="file-glob" paths="crates/supersigil-core/Cargo.toml" />
  </Criterion>
  <Criterion id="req-4-2">
    Evidence collection and proc-macro validation SHALL be in separate crates
    from `supersigil-core` so that the core model remains ecosystem-agnostic
    and the proc macro's heavy `syn` dependency is isolated from runtime code.
    <VerifiedBy strategy="file-glob" paths="crates/supersigil-evidence/Cargo.toml" />
  </Criterion>
  <Criterion id="req-4-3">
    THE evidence contract crate SHALL NOT import Rust-specific parsing types,
    so that future ecosystem plugins can implement the same trait surface
    without depending on the Rust toolchain.
    <VerifiedBy strategy="file-glob" paths="crates/supersigil-evidence/Cargo.toml" />
  </Criterion>
</AcceptanceCriteria>
