If you already completed the Quickstart, this walkthrough expands the same loop
into a fuller specification cycle: requirements, design links, evidence, and
graph queries. By the end you will have a connected spec graph with passing
verification.

## The scenario

You are building a user authentication feature. Users log in with credentials, and the system issues a session token. We will specify, design, and verify this feature.

## Step 1: Initialize the project

If you have not already, create a Supersigil project:

```bash
supersigil init
```

This creates `supersigil.toml` with the default spec path:

```toml title="supersigil.toml"
paths = ["specs/**/*.md"]
```

## Step 2: Write the requirements

Scaffold a requirements document:

```bash
supersigil new requirements auth
```

This creates `specs/auth/auth.req.md`. Open it and fill in actual requirements:

````md title="specs/auth/auth.req.md"
---
supersigil:
  id: auth/req
  type: requirements
  status: draft
title: "User Authentication"
---

## Introduction

Users authenticate via username and password. The system issues a session token
on success and rejects invalid credentials.

```supersigil-xml
<AcceptanceCriteria>
  <Criterion id="valid-creds">
    WHEN a user submits valid credentials
    THE SYSTEM SHALL issue a session token.
  </Criterion>

  <Criterion id="invalid-creds">
    WHEN a user submits invalid credentials
    THE SYSTEM SHALL return a 401 error.
  </Criterion>
</AcceptanceCriteria>
```

```supersigil-xml
<TrackedFiles paths="src/auth/**/*.rs" />
```
````

A few things to note about the structure:

- **Front matter** uses the `supersigil:` key with `id`, `type`, and `status`. The built-in document types are `requirements`, `design`, `tasks`, and `adr`.
- **`<AcceptanceCriteria>`** groups one or more `<Criterion>` components. It has no attributes of its own.
- **`<Criterion>`** defines a single verifiable criterion with a unique `id`. Other documents and tools reference criteria using the fragment syntax `doc-id#criterion-id` (e.g. `auth/req#valid-creds`).
- **`<VerifiedBy>`** can be nested inside a `<Criterion>` to declare explicit evidence using `strategy="tag"` (comment tags) or `strategy="file-glob"` (path patterns). For Rust projects, the ecosystem plugin discovers `#[verifies("doc#criterion")]` annotations automatically, so `<VerifiedBy>` is often unnecessary.
- **`<TrackedFiles>`** declares source file globs that this spec cares about. When those files change, `supersigil verify --since <ref>` flags the spec as stale.

<Aside type="note">
  Documents in `draft` status get lenient treatment: verification findings are
  downgraded to informational. This lets you iterate on spec structure without
  being blocked by missing test coverage.
</Aside>

### Valid statuses by document type

| Type | Valid statuses |
|------|---------------|
| `requirements` | `draft`, `review`, `approved`, `implemented` |
| `design` | `draft`, `review`, `approved` |
| `tasks` | `draft`, `ready`, `in-progress`, `done` |
| `adr` | `draft`, `review`, `accepted`, `superseded` |

## Step 3: Run verification

```bash
supersigil verify
```

Since the document is in `draft` status and no test files contain the tags yet, you will see a clean result with a hint:

```
✓ Clean: no findings
hint: 2 finding(s) downgraded to info because their documents have status: draft.
```

The 2 suppressed findings are `missing_verification_evidence` for each criterion -- no tests match the declared tags yet. Once you move the status out of `draft`, these become real errors.

## Step 4: Annotate your tests

Add annotations to the tests that prove each criterion. The ecosystem plugin for your language discovers these automatically.

**Rust** (plugin enabled by default):

```rust title="tests/auth/login_test.rs"
use supersigil_rust::verifies;

#[verifies("auth/req#valid-creds")]
#[test]
fn login_with_valid_credentials_returns_token() {
    let result = login("alice", "correct-password");
    assert!(result.token.is_some());
}
```

```rust title="tests/auth/login_failure_test.rs"
use supersigil_rust::verifies;

#[verifies("auth/req#invalid-creds")]
#[test]
fn login_with_wrong_password_returns_401() {
    let result = login("alice", "wrong-password");
    assert_eq!(result.status, 401);
}
```

Add `supersigil-rust` as a dev-dependency. The annotation is a no-op at runtime.

**JavaScript/TypeScript** (add `plugins = ["js"]` to config):

```typescript title="tests/auth/login.test.ts"
import { verifies } from '@supersigil/vitest'
import { test, expect } from 'vitest'

test('login with valid credentials', verifies('auth/req#valid-creds'), () => {
  const result = login('alice', 'correct-password')
  expect(result.token).toBeDefined()
})

test('login with wrong password returns 401', verifies('auth/req#invalid-creds'), () => {
  const result = login('alice', 'wrong-password')
  expect(result.status).toBe(401)
})
```

Add `@supersigil/vitest` as a dev-dependency.

<Aside type="tip">
  For projects without an ecosystem plugin, you can use `<VerifiedBy strategy="tag" tag="auth:valid-creds" />` in
  your criteria and add `// supersigil: auth:valid-creds` comments in test files instead.
  See the [Evidence Sources](/supersigil/concepts/evidence-sources/) page for all options.
</Aside>

## Step 5: Create a design document

Scaffold a design document for the same feature:

```bash
supersigil new design auth
```

Since `specs/auth/auth.req.md` already exists, the scaffold automatically includes a live `<Implements>` reference. Open `specs/auth/auth.design.md` and flesh it out:

````md title="specs/auth/auth.design.md"
---
supersigil:
  id: auth/design
  type: design
  status: draft
title: "Authentication Design"
---

```supersigil-xml
<Implements refs="auth/req" />
<TrackedFiles paths="src/auth/**/*.rs" />
```

## Overview

The authentication module validates credentials against a password hash store
and issues JWT session tokens on success.

## Architecture

The login flow:

1. `POST /api/login` receives username and password.
2. `AuthService::authenticate` verifies credentials against the store.
3. On success, `TokenService::issue` creates a signed JWT.
4. On failure, a 401 response is returned.

## Testing Strategy

Tag-based verification links tests to the requirement criteria.
Integration tests cover the full HTTP request/response cycle.
````

Key components used here:

- **`<Implements refs="auth/req" />`** declares that this design implements the `auth/req` requirements document. This creates an `implements` edge in the graph.
- **`<DependsOn>`** (not shown here) would declare dependencies on other documents. Supersigil uses these for cycle detection and dependency ordering.
- **`<References>`** (not shown here) creates informational traceability links. Unlike `Implements`, references do not affect verification coverage.

## Step 6: Verify the full graph

Now run verify again to check the entire graph:

```bash
supersigil verify
```

With test files containing the correct annotations, and both documents in `draft`:

```
✓ Clean: no findings
hint: 2 finding(s) downgraded to info because their documents have status: draft.
```

To see the real verification result, change the requirements status to `approved`:

```yaml
supersigil:
  id: auth/req
  type: requirements
  status: approved
```

Then run verify again. If the test files exist and contain the annotations:

```
✓ Clean: no findings
```

If the annotations are missing or the test files do not exist, you will see errors:

```
auth/req
  ✗ [missing_verification_evidence] Criterion "valid-creds": no matching evidence
  ✗ [missing_verification_evidence] Criterion "invalid-creds": no matching evidence

2 error(s), 0 warning(s), 0 info(s) across 2 documents
```

## What verification catches

When you run `supersigil verify`, it checks for:

- **Broken refs** -- references to document IDs or criterion IDs that do not exist.
- **Duplicate IDs** -- two documents with the same `supersigil.id`, or two criteria with the same ID in one document.
- **Uncovered criteria** -- criteria with no verification evidence (no matching `VerifiedBy` tags or file globs, no plugin evidence).
- **Cycles** -- circular `DependsOn` or task dependency chains.
- **Stale tracked files** -- when files matched by `TrackedFiles` have changed since a git ref (requires `--since`).
- **Missing test files** -- `VerifiedBy strategy="file-glob"` patterns that match zero files.
- **Zero tag matches** -- `VerifiedBy strategy="tag"` tags not found in any test file.
- **Orphan test tags** -- `// supersigil: <tag>` comments in test files that are not declared by any `VerifiedBy`.
- **Invalid VerifiedBy placement** -- `VerifiedBy` components placed outside of a verifiable parent (like `Criterion`).
