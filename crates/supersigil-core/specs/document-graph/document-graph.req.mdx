## Introduction

This spec recovers the current `supersigil-core::graph` behavior: graph
construction from parsed documents plus config, structural validation, derived
indexes, and structured query outputs for `context` and `plan`.

This document replaces the stale root-level graph spec with the current query
model. The graph no longer carries illustration fields, and `References` links
are informational at graph-query time rather than direct verification evidence.

## Definitions

- **Graph_Builder**: The `build_graph` pipeline in `supersigil-core::graph`.
- **Document_Index**: The `doc_index` keyed by document ID.
- **Component_Index**: The `(document_id, fragment)` index for referenceable
  components.
- **Resolved_Ref**: A resolved doc or doc-fragment reference captured during
  graph construction.
- **Project_Membership**: The optional project assigned to a document in
  multi-project mode.
- **Verifiable_Target**: A referenceable component whose definition has
  `verifiable = true`.

## Requirement 1: Indexing and Project Membership

As a graph consumer, I want documents and referenceable components indexed
before relationship analysis, so that later stages can resolve refs in
constant-time lookups.

<AcceptanceCriteria>
  <Criterion id="req-1-1">
    THE Graph_Builder SHALL index each parsed document by `frontmatter.id`. IF
    two or more documents share an ID, THEN graph construction SHALL report
    `GraphError::DuplicateId`.
    <VerifiedBy strategy="file-glob" paths="crates/supersigil-core/src/graph/tests/prop_document_index.rs, crates/supersigil-core/src/graph/tests/prop_component_index.rs, crates/supersigil-core/src/graph/index.rs" />
  </Criterion>
  <Criterion id="req-1-2">
    THE Graph_Builder SHALL recursively index every referenceable component by
    `(document_id, fragment)`, including nested components. IF the same
    fragment appears more than once within a document, THEN graph construction
    SHALL report `GraphError::DuplicateComponentId`.
    <VerifiedBy strategy="file-glob" paths="crates/supersigil-core/src/graph/tests/prop_component_index.rs" />
  </Criterion>
  <Criterion id="req-1-3">
    IN Multi_Project_Mode, THE Graph_Builder SHALL assign Project_Membership to
    each indexed document based on the configured project path prefixes. IN
    Single_Project_Mode, documents SHALL have no project assignment.
    <VerifiedBy strategy="file-glob" paths="crates/supersigil-core/src/graph/tests/prop_ref_resolution.rs" />
  </Criterion>
</AcceptanceCriteria>

## Requirement 2: Ref Resolution and Structural Validation

As a graph consumer, I want refs validated against the indexed documents and
components, so that downstream queries only operate on structurally consistent
graphs.

<AcceptanceCriteria>
  <Criterion id="req-2-1">
    FOR every component with a `refs` attribute, THE Graph_Builder SHALL split
    the raw value with `split_list_attribute` and resolve each entry against
    the Document_Index. Fragment refs SHALL additionally resolve against the
    Component_Index.
    <VerifiedBy strategy="file-glob" paths="crates/supersigil-core/src/graph/resolve.rs" />
  </Criterion>
  <Criterion id="req-2-2">
    IF a resolved fragment is constrained by `target_component`, THEN the
    target component name SHALL match that constraint. If the document,
    fragment, or expected component type does not match, THEN graph
    construction SHALL report `GraphError::BrokenRef`.
    <VerifiedBy strategy="file-glob" paths="crates/supersigil-core/src/graph/tests/prop_ref_resolution.rs" />
  </Criterion>
  <Criterion id="req-2-3">
    Task `implements` refs SHALL be resolved separately from generic `refs`.
    Each `implements` ref SHALL include a fragment and SHALL target a
    Verifiable_Target. IF it does not, THEN graph construction SHALL report
    `GraphError::BrokenRef`.
    <VerifiedBy strategy="file-glob" paths="crates/supersigil-core/src/graph/tests/prop_task_implements.rs" />
  </Criterion>
  <Criterion id="req-2-4">
    WHEN a source document belongs to a non-isolated project, ref resolution
    SHALL use the workspace-wide document graph. WHEN a source document belongs
    to an isolated project, refs into a different project SHALL fail as broken
    refs while same-project refs continue to resolve.
    <VerifiedBy strategy="file-glob" paths="crates/supersigil-core/src/graph/tests/prop_ref_resolution.rs" />
  </Criterion>
</AcceptanceCriteria>

## Requirement 3: Dependency Analysis and Ordering

As a query consumer, I want task and document dependencies analyzed once at
graph-build time, so that query output can present work in deterministic order.

<AcceptanceCriteria>
  <Criterion id="req-3-1">
    THE Graph_Builder SHALL resolve task `depends` edges within sibling scope:
    top-level tasks depend on top-level siblings, and nested tasks depend on
    siblings under the same parent. Missing task dependencies SHALL produce
    broken-ref errors.
    <VerifiedBy strategy="file-glob" paths="crates/supersigil-core/src/graph/tests/prop_cycle.rs, crates/supersigil-core/src/graph/tests/prop_topo.rs" />
  </Criterion>
  <Criterion id="req-3-2">
    THE Graph_Builder SHALL detect cycles in task dependencies and in
    document-level `DependsOn` edges, including self-cycles, and SHALL report
    them as hard graph errors.
    <VerifiedBy strategy="file-glob" paths="crates/supersigil-core/src/graph/tests/prop_cycle.rs" />
  </Criterion>
  <Criterion id="req-3-3">
    WHEN no cycle errors are present, THE Graph_Builder SHALL compute
    deterministic topological orderings for tasks and documents. Task ordering
    SHALL break ties by declaration order; document ordering SHALL break ties
    alphabetically by document ID.
    <VerifiedBy strategy="file-glob" paths="crates/supersigil-core/src/graph/tests/prop_topo.rs" />
  </Criterion>
</AcceptanceCriteria>

## Requirement 4: Derived Indexes

As a downstream consumer, I want reverse mappings and tracked-file indexes
precomputed from the clean graph, so that verification and query layers do not
need to rescan component trees.

<AcceptanceCriteria>
  <Criterion id="req-4-1">
    THE Graph_Builder SHALL compute reverse mappings for `References`,
    `Implements`, and `DependsOn`. `References` SHALL retain fragmentless refs
    as `(doc_id, None)` keys. `Implements` SHALL discard fragment detail in the
    reverse mapping even if the source ref used a fragment.
    <VerifiedBy strategy="file-glob" paths="crates/supersigil-core/src/graph/tests/prop_reverse.rs, crates/supersigil-core/src/graph/tests/prop_tracked_files.rs, crates/supersigil-core/src/graph/reverse.rs" />
  </Criterion>
  <Criterion id="req-4-2">
    Duplicate ref strings inside one attribute SHALL contribute only once to the
    reverse mappings.
    <VerifiedBy strategy="file-glob" paths="crates/supersigil-core/src/graph/tests/prop_reverse.rs" />
  </Criterion>
  <Criterion id="req-4-3">
    THE Graph_Builder SHALL aggregate `TrackedFiles` `paths` values by owning
    document, expose lookup by document ID, and expose iteration across all
    tracked-file entries.
    <VerifiedBy strategy="file-glob" paths="crates/supersigil-core/src/graph/tests/prop_tracked_files.rs" />
  </Criterion>
</AcceptanceCriteria>

## Requirement 5: Structured Queries

As an agent or developer, I want structured `context` and `plan` outputs from a
clean graph, so that implementation planning does not depend on screen-scraped
terminal output.

<AcceptanceCriteria>
  <Criterion id="req-5-1">
    `context(id)` SHALL return the target document, its `Criterion` components
    as `TargetContext` values, optional-status `implemented_by` documents,
    document-level `referenced_by` docs, and linked tasks in topological order.
    Linked tasks SHALL carry resolved `implements` mappings and `depends_on`
    task IDs.
    <VerifiedBy strategy="file-glob" paths="crates/supersigil-core/src/graph/tests/prop_context.rs, crates/supersigil-core/src/graph/tests/prop_plan.rs, crates/supersigil-core/src/graph/query.rs, crates/supersigil-core/tests/serialize_roundtrip.rs" />
  </Criterion>
  <Criterion id="req-5-2">
    `plan` SHALL support exact-document, prefix, and all-documents queries.
    Query parsing SHALL prefer an exact document match, then a prefix match,
    and otherwise report no matching documents.
    <VerifiedBy strategy="file-glob" paths="crates/supersigil-core/src/graph/tests/prop_plan.rs" />
  </Criterion>
  <Criterion id="req-5-3">
    `plan` SHALL report `outstanding_targets` for verifiable criteria in the
    target document set that are not implemented by a completed task.
    `References` links SHALL remain informational at this layer and SHALL NOT
    suppress outstanding targets.
    <VerifiedBy strategy="file-glob" paths="crates/supersigil-core/src/graph/tests/prop_plan.rs" />
  </Criterion>
  <Criterion id="req-5-4">
    `plan` SHALL return linked pending and completed tasks in topological
    order, preserving each task's resolved `implements` data.
    <VerifiedBy strategy="file-glob" paths="crates/supersigil-core/src/graph/tests/prop_plan.rs" />
  </Criterion>
  <Criterion id="req-5-5">
    `ContextOutput` and `PlanOutput` SHALL remain structured serializable types
    rather than preformatted strings.
    <VerifiedBy strategy="file-glob" paths="crates/supersigil-core/tests/serialize_roundtrip.rs" />
  </Criterion>
</AcceptanceCriteria>

## Requirement 6: Error Aggregation

As a graph consumer, I want one build pass to surface as many structural
problems as possible, so that I do not have to fix graph failures one at a
time.

<AcceptanceCriteria>
  <Criterion id="req-6-1">
    THE Graph_Builder SHALL aggregate duplicate-ID, broken-ref, and dependency
    errors discovered across indexing, resolution, and cycle-detection stages
    into one returned error collection.
    <VerifiedBy strategy="file-glob" paths="crates/supersigil-core/src/graph/tests/prop_error_aggregation.rs, crates/supersigil-core/src/graph.rs" />
  </Criterion>
  <Criterion id="req-6-2">
    IF any hard graph errors are present, THEN `build_graph` SHALL return the
    collected errors instead of a partial graph. Derived indexes such as topo
    orders, reverse mappings, and tracked files SHALL be computed only for a
    clean graph.
    <VerifiedBy strategy="file-glob" paths="crates/supersigil-core/src/graph/tests/prop_error_aggregation.rs" />
  </Criterion>
</AcceptanceCriteria>
