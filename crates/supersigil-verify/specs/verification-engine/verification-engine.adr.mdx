## Context

Supersigil verifies that specifications are internally consistent and
that code stays aligned with specs. The verification model must balance
strictness (catching real problems) with usability (not blocking
iterative authoring or penalizing teams with different practices).

## Decisions

<Decision id="spec-drift-three-signals">
  Supersigil addresses spec drift with three mechanisms at different
  confidence levels: `<TrackedFiles>` (routing signal), ecosystem
  annotations (evidence signal), and executable examples (proof signal).

  <References refs="verification-engine/req#req-6-1, verification-engine/req#req-6-2, verification-engine/req#req-4-3, verification-engine/req#req-3-1" />

  <Rationale>
    Each level trades effort for confidence. `<TrackedFiles>` is cheap
    and broad — it says "review this spec" when associated files change.
    Ecosystem annotations (e.g. `#[verifies("doc#criterion")]`) are
    precise but require annotating tests. Executable examples are the
    strongest signal but require maintaining inline code. Teams pick the
    level that matches their rigor needs without an all-or-nothing choice.
  </Rationale>

  <Alternative id="test-derived-file-associations" status="rejected">
    Deriving file associations from test mappings (if a test file
    changes, the spec it verifies is potentially stale). Rejected because
    tests and specs can address different aspects of the same code, and
    test file changes don't always indicate spec drift.
  </Alternative>
</Decision>

<Decision id="configurable-strictness">
  Findings are split into hard errors (always fatal, never configurable)
  and configurable rules with overridable severities. Four levels of
  precedence: draft gating, per-rule overrides, global strictness,
  built-in defaults.

  <References refs="verification-engine/req#req-2-1, verification-engine/req#req-2-4" />

  <Rationale>
    Hard errors are structural integrity failures — broken refs, duplicate
    IDs, missing required attributes, expression attributes, dependency
    cycles. Suppressing them invites real problems, so making them
    unconfigurable eliminates the temptation. For everything else, one
    severity vocabulary across all scopes means zero translation overhead.
    Unknown rule keys in `[verify.rules]` are config errors to catch
    typos.
  </Rationale>

  <Alternative id="all-configurable" status="rejected">
    Making hard errors configurable would allow users to suppress broken
    refs or duplicate IDs. This creates a false sense of safety — the
    graph is structurally invalid, and downstream tools will produce
    unpredictable results.
  </Alternative>
</Decision>

<Decision id="status-gating">
  Documents with `status: draft` have all configurable rules suppressed
  to `info` level. Findings still appear in the output but don't fail
  the build. Hard errors are never suppressed.

  <References refs="verification-engine/req#req-2-2" />

  <Rationale>
    This solves the tension between strict defaults and iterative
    authoring. `status: draft` makes strictness humane — you write the
    spec incrementally, and supersigil tells you what's missing without
    blocking you. When you promote the status, the full rule set applies.
  </Rationale>

  <Alternative id="no-draft-gating" status="rejected">
    Without draft gating, authors must either satisfy all rules before
    committing (discourages incremental work) or configure rules to be
    lenient globally (weakens verification for finished specs). Neither
    is acceptable.
  </Alternative>
</Decision>

<Decision id="advisory-status">
  Statuses (`draft`, `approved`, `verified`, etc.) are informational.
  Supersigil reports inconsistencies (e.g. `status: verified` with no
  tests) but does not prevent status transitions.

  <References refs="verification-engine/req#req-5-4" />

  <Rationale>
    Enforcing state machines in a CLI tool creates friction without
    proportional value. A status field that cannot be freely edited
    requires workflow tooling (who can approve? what are the
    preconditions?) that belongs in a project management layer, not a
    verification tool. Advisory status gives teams the vocabulary to
    communicate intent while keeping the tool lightweight.
  </Rationale>

  <Alternative id="enforced-state-machines" status="rejected">
    Blocking status transitions that violate preconditions (e.g. cannot
    move to `verified` without passing tests) would require supersigil
    to model team workflows. Different teams have different gates, and
    hardcoding any particular state machine would be too rigid for some
    and too lenient for others.
  </Alternative>
</Decision>

## Consequences

Teams get strict structural validation out of the box with no
configuration. Quality rules can be tuned per-project. Draft specs are
never blocked, so iterative authoring is safe. The cost is that some
rules require teams to explicitly opt in (`missing_decision_coverage`,
`isolated_document`), which means they may not discover the value until
they look for it.
